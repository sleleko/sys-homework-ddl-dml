# Домашнее задание к занятию «Резервное копирование баз данных» - `Сергей Лелеко`

### Задание 1 Резервное копирование
#### 1.1. Восстановление данных в полном объёме за предыдущий день

Для этой задачи подходит ежедневное **полное резервное** копирование.

Например:

Каждый день в 02:00 → полный backup базы данных
Хранение копий → 7 / 14 / 30 дней
При сбое -> восстановление последней полной копии

`Если база сломалась сегодня, можно восстановить состояние базы на момент последнего полного backup, например за прошлую ночь.`

Плюсы:

    простая настройка;

    простое восстановление;

    понятный контроль наличия резервных копий.

Минусы:

`можно потерять данные, которые появились после последнего backup;
для финансовой компании одного полного backup обычно недостаточно.`

#### 1.2. Восстановление данных за час до предполагаемой поломки

Для этой задачи нужен не только полный backup, но и сохранение журналов изменений.

Для PostgreSQL 17 это может быть:

`полный backup + архивирование WAL`

PostgreSQL 17 хранит изменения в WAL — write-ahead log. Документация PostgreSQL 17 описывает, что сочетание резервной копии файлового уровня и архивных WAL-файлов позволяет выполнить Point-in-Time Recovery, то есть восстановление на конкретный момент времени. ([PostgreSQL](https://www.postgresql.org/docs/17/continuous-archiving.html))

Для MySQL 8.0 это может быть:

`полный backup + binary logs`

Документация MySQL 8.0 описывает восстановление на момент времени через binary log: после восстановления полного backup применяются события из binary log до нужного момента. Для такого восстановления binary logging должен быть включён; в MySQL 8.0 он включён по умолчанию. ([dev.mysql.com](https://dev.mysql.com/doc/mysql-backup-excerpt/8.0/en/point-in-time-recovery-binlog.html))

Пример логики:

    02:00 -> полный backup
    Далее -> сохраняются WAL / binary logs
    15:00 -> обнаружена поломка
    Восстановление → полный backup + журналы изменений до 14:00

Такой подход лучше подходит для финансовой компании, потому что позволяет уменьшить потерю данных.

#### 1.3.* Моментальное переключение на работающую базу
Да, такой вариант возможен, но это уже не просто резервное копирование, а отказоустойчивая архитектура.

Пример:

    Основной сервер БД → принимает запись и чтение
    Реплика / standby-сервер → получает изменения с основного сервера
    При сбое → приложение переключается на реплику

Для PostgreSQL 17 можно использовать streaming replication и standby-сервер. Реплика может быть переведена в режим основного сервера при отказе primary-сервера. Но важно понимать: репликация не заменяет backup. Если пользователь случайно удалит данные, ошибка может быть передана и на реплику.

Правильная схема:

    Резервное копирование → защита от потери и повреждения данных
    Репликация → быстрое переключение при аварии
    Backup + Replica → более надёжная промышленная схема

### Задание 2. PostgreSQL 17
#### 2.1. Пример команды резервирования и восстановления БД через pg_dump / pg_restore
В PostgreSQL 17 для логического резервного копирования можно использовать `pg_dump`.

Официальная документация PostgreSQL 17 указывает, что `pg_dump` может создавать backup в разных форматах. Формат `custom`, то есть `-Fc`, подходит для восстановления через `pg_restore`, поддерживает выборочное восстановление и сжатие по умолчанию. ([PostgreSQL](https://www.postgresql.org/docs/17/app-pgdump.html))

**Создание резервной копии PostgreSQL 17**

```sql
pg_dump -U postgres -h localhost -p 5432 -F c -b -v -f /backup/mydb.dump mydb
```

Где:
```sql
-U postgres          пользователь PostgreSQL
-h localhost         адрес сервера
-p 5432              порт PostgreSQL
-F c                 custom-формат backup
-b                   включить large objects
-v                   подробный вывод
-f                   файл резервной копии
mydb                 имя базы данных
```
**Восстановление в новую базу**

Сначала создаём пустую базу:
```sql
createdb -U postgres -h localhost -p 5432 -T template0 mydb_restore
```
Затем восстанавливаем backup:
```sql
pg_restore -U postgres -h localhost -p 5432 -d mydb_restore -v /backup/mydb.dump
```
Документация PostgreSQL 17 показывает аналогичный вариант: сначала создаётся база через `createdb`, затем архив восстанавливается командой `pg_restore -d newdb db.dump`. ([PostgreSQL](https://www.postgresql.org/docs/17/app-pgrestore.html))

**Восстановление с пересозданием базы**

Можно использовать вариант, когда база создаётся из backup:
```sql
pg_restore -U postgres -h localhost -p 5432 -C -d postgres -v /backup/mydb.dump
```

Здесь параметр `-C` означает, что `pg_restore` создаст базу данных из дампа. В документации PostgreSQL 17 также приведён пример восстановления с пересозданием базы через `pg_restore -C -d postgres db.dump`. ([PostgreSQL](https://www.postgresql.org/docs/17/app-pgrestore.html))

### Задание 3. MySQL 8.0
#### 3.1. Пример команды инкрементного резервного копирования MySQL 8.0
Можно использовать схему:

    полный backup через mysqldump
    +
    binary logs для восстановления изменений

Пример полного backup:

```shell 
mysqldump -u root -p \
--single-transaction \
--routines \
--triggers \
--events \
--databases mydb \
> /backup/mysql/mydb_full.sql
```

Пример восстановления полного backup:

```shell
mysql -u root -p < /backup/mysql/mydb_full.sql
```

Затем можно применить binary logs:

```shell
mysqlbinlog /var/lib/mysql/binlog.000001 /var/lib/mysql/binlog.000002 | mysql -u root -p
```

Для восстановления до конкретного времени можно использовать `--stop-datetime`:
```shell
mysqlbinlog \
--stop-datetime="2026-05-25 14:00:00" \
/var/lib/mysql/binlog.000001 /var/lib/mysql/binlog.000002 \
| mysql -u root -p
```
Это не совсем "инкрементный backup" в смысле отдельной инкрементной копии, но это официальный способ восстановления MySQL 8.0 на момент времени с использованием binary log. ([dev.mysql.com](https://dev.mysql.com/doc/mysql-backup-excerpt/8.0/en/point-in-time-recovery-binlog.html))

#### 3.1.* Когда реплика MySQL 8.0 лучше обычного backup?

Реплика MySQL 8.0 даёт преимущество, когда важно быстро восстановить работу сервиса.

Обычный backup нужно сначала найти, развернуть и проверить. На это может уйти много времени. Реплика же уже содержит актуальную или почти актуальную копию данных.

Преимущества реплики:

    быстрое переключение при сбое основного сервера;
    разгрузка основного сервера за счёт SELECT-запросов на реплике;
    возможность делать backup с реплики;
    меньшее время простоя;
    удобство для отчётов и аналитических запросов.

Документация MySQL 8.0 описывает сценарий, когда backup выполняется с реплики, чтобы не останавливать и не нагружать основной source-сервер. Реплику можно временно остановить или приостановить для получения согласованного snapshot. ([dev.mysql.com](https://dev.mysql.com/doc/mysql-backup-excerpt/8.0/en/point-in-time-recovery-binlog.html))

**Но реплика не заменяет резервное копирование.**

Например:
```sql
DROP TABLE payments;
```

Если такая команда была ошибочно выполнена на основном сервере, она может попасть и на реплику. Поэтому реплика защищает от отказа сервера, но не всегда защищает от логических ошибок пользователя.

Правильная схема для MySQL 8.0:

    полный backup;
    инкрементный backup или binary logs;
    репликация;
    проверка восстановления;
    хранение копий отдельно от основного сервера.

backup нужен для восстановления данных, а репликация — для высокой доступности. В надёжной системе используются оба подхода одновременно.
